剪贴板富文本保真
实现方案

让 QuickQuick 复制带格式内容时捕获并保存 HTML，预览区可视化呈现，粘贴 / 复制回系统时还原格式。核心发现：arboard 3.6.1 三平台已原生支持 HTML 读写，跨平台几乎免费。

分支建议　feat/clipboard-richtext-fidelity 制定　2026-06-07 状态　待用户批准后进入 Phase 5
一

目标与范围

让剪贴板历史保真富文本：复制带格式内容（加粗 / 颜色 / 列表 / 表格 / 链接等）时捕获并保存其 HTML，预览可视化呈现，粘贴 / 复制回系统时还原格式。

已冻结的范围决策（用户拍板）

#决策点选定
1格式范围只 HTML（不做 RTF）。覆盖浏览器 / Office / 多数富文本源；纯 RTF 源退纯文本
2还原范围「粘贴到前台」与「复制」按钮都带富文本
3预览渲染预览区渲染真富文本 + DOMPurify 清洗；列表行仍纯文本摘要
4去重语义纯文本相同即去重；命中旧行但新来的带 html 而旧行没有，则补写 html 并升为 richtext

明确不做（YAGNI）

  • RTF 捕获 / 还原　arboard 不支持，需各平台原生 + cfg 分支，成本高
  • HTML 内联远程图片的下载 / 内嵌、完整 CSS 保真　取决于目标 app 渲染能力
  • 列表行的富文本缩略渲染　保持纯文本摘要，性能与可读性更好
二

关键发现（决定复杂度）

arboard 3.6.1 三平台均已原生实现 HTML 的「读」和「写」——无需平台原生代码、无需新增依赖、无需 #[cfg(target_os)] 分支。

平台读 HTML写 HTMLarboard 源码位置
macOSstring_from_type(NSPasteboardTypeHTML)setString_forTypeplatform/osx.rs:212 / 301
Windowsclipboard_win::raw::get_htmlwrap_html + set（自动处理 CF_HTML 偏移头）platform/windows.rs:611 / 688
LinuxX11 text/html atom / Wayland mime同左platform/linux/{x11,wayland}.rs
调用方式

读：clipboard.get().html().ok()　|　写：clipboard.set().html(html, Some(plain_text_alt))（同时写 HTML 与纯文本兜底）。

这推翻了"跨平台需手撸 NSPasteboard / CF_HTML / Wayland"的预设——跨平台 HTML 几乎免费。

现状根因（富文本为何丢失）

数据模型早已预留 html 字段，但三处把它截断：

捕获

pipeline.rs:155

ArboardBackend::read() 中 html: None 硬编码；便捷方法 get_text() 只读纯文本。

存储

db.rs:471

clip_items 表无 html 列；ingest() 的 INSERT 只写 6 字段、kind 写死 'text'。

还原

system.rs:141 / macos_paste.rs:91

fetch_paste_item 构造 html:None；write_with_marker 只 set_text。

前端

已为 richtext 预留

kind: "text" | "richtext" | "image" 贯穿搜索 / 过滤 / 历史，连"富文本"筛选 tab 都在——本方案是接通已铺好的轨道。

四

改动清单（按 TDD 任务顺序）

每个任务红-绿-重构。后端 Rust 用 coder(Opus)，测试 tester(Sonnet) 动态证伪，规范审查 code-reviewer(Sonnet)。每个任务测试通过方进下一个。

任务 1

存储层：html 列 + ingest + 查询

db.rs
  • ensure_schema 建表加 html_content TEXT；存量库用 PRAGMA table_info 守卫 + ALTER TABLE ADD COLUMN 幂等迁移
  • ingest：INSERT 增 html_content，kind = html 有则 'richtext'；去重命中且旧行无 html、新行有 → UPDATE 补写再 bump；text_hash 不变
  • 查询行结构 + SELECT 补 html_content
测试：roundtrip / 迁移 / 补写 / 去重不变
任务 2

捕获层：读 HTML + 变化检测

pipeline.rs
  • read()：html: cb.get().html().ok()
  • compute_composite_hash 把 html 字节拼入（独立分隔符、顺序固定），使"同纯文本但新增 / 变更 html"触发 change_count → 走 ingest 补写
测试：html 不同 → hash 不同
任务 3

IPC 取数：DTO + 前端类型

ipc/clipboard.rs · ipc-client.ts
  • ClipItemDto 加 html_content: Option<String>（camelCase → htmlContent）
  • TS ClipItem 加 htmlContent?: string
测试：富文本行返回 html_content，纯文本 null
任务 4

还原：粘贴 + 复制

system.rs · macos_paste.rs
  • fetch_paste_item SELECT + 构造 CapturedItem{text,html}
  • write_with_marker（macos_impl + fallback_impl）：有 html 走 set().html(h, Some(text))，否则 set_text；trait 签名不变
  • 新增 IPC copy_clip_to_clipboard(id) 走 arboard set().html，前端"复制"改调它
测试：fetch 取出 html；复制命令取数 + 组装
任务 5

前端渲染 + 安全清洗

ClipPreview · PopoverPreview
  • 新增 dompurify（+ types）
  • kind==='richtext' && htmlContent → dangerouslySetInnerHTML={{__html: DOMPurify.sanitize(htmlContent)}}；否则纯文本
  • popover 预览同步；列表行不变
测试：<script>/onerror 被剥离，正常格式保留
五

安全（XSS 红线）

引入 dangerouslySetInnerHTML 是本方案唯一显著风险面。硬约束：

  • 任何 html 入 DOM 前必须 DOMPurify.sanitize　无例外、无"信任来源"豁免（剪贴板来源不可信）
  • 白名单只放安全展示标签 / 属性　剥离 <script>、事件处理器、javascript: URL、<iframe>/<object> 等
  • 后端原样保存未清洗 html（保真）　清洗只在渲染层——保证粘贴 / 复制回去拿到的是用户原始格式
  • CSP 复核　预览渲染富文本不得放开 script-src
六

测试策略（动态证伪 · 硬门禁）

  • Rust 单测　迁移幂等、ingest / 补写 / 去重、fetch_paste_item、hash 区分 html
  • 前端单测　DOMPurify 剥离恶意标签、保留正常格式、纯文本回退
  • tester 变异 sanity　注释掉 sanitize 应令安全测试失败；把 html 读取改常量应令 roundtrip 失败
  • 真机视觉　受屏幕录制权限限制，交用户截图确认（见记忆 quickquick-gui-verify-env）
  • 完成定义　lint / tsc / cargo check 通过、无遗留 TODO、符合 code-standards 与项目规范
七

风险与回退

风险应对
存量加密库加列失败PRAGMA table_info 守卫 + ADD COLUMN（NULL 默认）低风险；失败仅记录日志、退纯文本，不阻断启动
某些源 HTML 含大量内联样式 / 碎片（如 Word CF_HTML 片段）保真保存；预览靠 DOMPurify 容错渲染；不追求像素级还原
粘贴到不支持 HTML 的目标 appset().html(h, Some(text)) 已带纯文本兜底，目标 app 自动取纯文本
前端 XSS见第五节硬约束
与并发后端工作冲突对侧 ECDICT 重构已收尾、链路不重叠；开工前 git pull / rebase 同步
