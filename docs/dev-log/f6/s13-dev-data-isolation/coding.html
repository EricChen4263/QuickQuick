dev/release 数据目录隔离
debug 落 dev 子目录

正式版装机后剪贴板报「加载失败」——根因是 debug 与 release 共用数据目录、密钥体系不同导致 SQLCipher 解密失败。本方案在路径解析层单点收口，让 debug 构建落 dev 子目录，与 release 彻底隔离。

层级　小功能 · parent F6 制定　2026-06-04 状态　待实现 → 交 coder
一

要解决的 bug

正式版（release，adhoc 未签名）装机后，剪贴板页报「加载失败，请稍后重试」。下列复现与定位已实测坐实，非推断。

  • 从终端带日志启动 release 二进制，stderr：数据库打开失败：数据库操作失败：file is not a database
  • file is not a database 是 SQLCipher 密钥不匹配时的典型表现：头部解密不出来，遂报「不是数据库」。
  • 数据目录 ~/Library/Application Support/com.quickquick.app/ 同时存在 dev-master-key、dev-credentials.json（debug 文件密钥库产物）与 quickquick.db——该 DB 是先前 dev 构建用文件密钥加密的。
  • release 走 keychain 密钥路径（KeychainKeyProvider），用 keychain 派生密钥去开 dev 文件密钥加密的库 → 不匹配 → 开库失败。
  • 验证：把数据目录整体挪开后，release 从零建库（217KB 全新 quickquick.db，无 dev 文件、无报错），剪贴板正常。证明 release 对全新用户无问题、发布有效；问题仅在「同机 dev/release 共用数据目录」时出现。
二

根因

debug 与 release 构建共用同一 identifier com.quickquick.app → 同一数据目录 app_config_dir()。两者密钥体系不同：

debug
FileKeyProvider · 文件 dev-master-key
release
KeychainKeyProvider · macOS 钥匙串
撞车
后跑方用己方密钥开对方加密的 DB
表象
file is not a database

用户只装 release 永不触发；开发者每次切构建就踩。

三

方案（单点收口，DRY）

所有持久化文件路径都源自 app.path().app_config_dir()。中心 helper resolve_config_dir()（ipc/settings.rs:234）已服务 ipc 层，但 lib.rs 有 5 处直接调 app_config_dir() 绕过了它。

改动 1 · resolve_config_dir() 加 debug 子目录

取得 app_config_dir() 后，#[cfg(debug_assertions)] 追加 dev 子目录；release 不追加。create_dir_all 对最终目录执行。

pub(crate) fn resolve_config_dir(app: &AppHandle) -> Result<PathBuf, String> { let base = app.path().app_config_dir() .map_err(|e| format!("无法获取配置目录：{e}"))?; // debug 落 dev 子目录，与 release（钥匙串密钥）彻底隔离， // 消除同机切换时 file is not a database。 #[cfg(debug_assertions)] let dir = base.join("dev"); #[cfg(not(debug_assertions))] let dir = base; std::fs::create_dir_all(&dir) .map_err(|e| format!("无法创建配置目录：{e}"))?; Ok(dir) }

改动 2 · lib.rs 5 处直接调用全部改走 resolve_config_dir()

位置用途现状改为
lib.rs:192 setup_app_dbquickquick.db 开库目录app_config_dir() + 手动 create_dir_allresolve_config_dir(app.handle())
lib.rs:282 捕获轮询读 settings.json 取 max_image_bytes直接 app_config_dir()同上
lib.rs:319 autostart 同步autostart.json直接 app_config_dir()同上
lib.rs:458 热键读取hotkey.json直接 app_config_dir()同上
lib.rs:491 设置布尔字段读取settings.json直接 app_config_dir()同上
  • setup_app_db(app: &mut tauri::App) 取 handle 用 app.handle()（&AppHandle），传给 resolve_config_dir。
  • 各处原有「拿不到目录则 eprintln 不 panic / 跳过」的降级语义必须保留；返回 Err 时走原降级路径，不得改成 panic。
  • 改后 lib.rs 不再直接 use app_config_dir（未用 import 一并清理）。
四

不做什么（边界）

B1

不改 identifier

tauri.conf.json 的 identifier 是构建期静态值，debug/release 同一份 conf 无法分流；路径层加子目录是最小且正确的做法。

B2

不动 release 路径

release 仍落 app_config_dir() 根，已装真实用户零迁移、零影响。

B3

不做 dev 旧库迁移

dev 是测试数据，落新 dev 子目录后旧根目录的 dev 文件自然弃用，无需搬迁。

B4

不改单测

keyprovider/credential 单测用 tempdir 显式传目录，与本改动正交。

五

验收标准

#验收项
1make verify 五步全绿（tsc / cargo fmt --check / clippy -D warnings / vitest / cargo test）。
2debug 构建运行后，所有文件（quickquick.db / dev-master-key / dev-credentials.json / settings.json / hotkey.json）落在 <app_config_dir>/dev/；根目录不再新增这些文件。
3release 路径不变：文件仍落 <app_config_dir>/ 根（cargo build --release 跑一次或审 cfg 分支确认）。
4同机先跑 release（根目录建库）再跑 debug（dev 子目录建库），两者互不报 file is not a database。
5lib.rs 5 处降级语义（拿不到目录不 panic）保持不变。
TDD 说明

目录解析是既有注释明示的「不可单测的胶水层」（依赖 Tauri AppHandle 运行时）。本 slice 以改动小、cfg 对称、make verify + 手动落盘验证为主要保障。若可低成本抽出「给定 base 返回带/不带 dev 后缀」的纯函数则加单测，但不为凑测试强行过度抽象。
